## CRITICAL

### C1 — Accessibility: Labels not associated with inputs (79 of 80 labels missing `htmlFor`)

**File:** `components/ExpenseModal.tsx`, `components/IncomeModal.tsx`, `components/AllocationModal.tsx`, `components/DebtModal.tsx`, `components/AdhocExpenseModal.tsx`, `components/AdhocIncomeModal.tsx`, `components/AdhocAllocationModal.tsx`, `components/ProjectModal.tsx`, `components/SectionsClient.tsx`, `components/CartesClient.tsx`, `components/TransferSavingsModal.tsx`, `components/AddSavingsModal.tsx`, `components/RevenusTrackingClient.tsx`, `components/ProjetsEpargneClient.tsx`
**Rule:** `af-frontend-clean-code` § Accessibilite rule 3 — "Labels sur les inputs: `<label htmlFor>` explicite"
**Problem:** Out of 80 `<label>` elements across the codebase, only 1 has an `htmlFor` attribute (`DebtModal.tsx`). All form inputs lack programmatic association with their labels. Screen readers cannot connect labels to their respective inputs, making forms unusable for assistive technology users.
**Fix:** Add `id` attributes to all `<input>`, `<select>`, and `<textarea>` elements, then add matching `htmlFor` attributes on their `<label>` elements. Example: `<label htmlFor="expense-name">` + `<input id="expense-name" />`.

### C2 — Accessibility: `userScalable: false` prevents zoom

**File:** `app/layout.tsx:35`
**Rule:** `af-frontend-clean-code` § Accessibilite rule 6 + WCAG 1.4.4
**Problem:** `userScalable: false` in the viewport meta tag prevents users from zooming on the page. This is a WCAG violation that affects users with low vision who need to enlarge content to read it. Combined with `maximumScale: 1`, it completely locks the zoom level.
**Fix:** Remove `userScalable: false` and change `maximumScale` to at least `5`. The PWA experience is preserved without blocking zoom.

### C3 — Accessibility: Interactive `<div onClick>` used for sheet backdrops without keyboard support

**File:** `components/DepensesTrackingClient.tsx:511`, `components/AllocationModal.tsx:91`, `components/AdhocAllocationModal.tsx:52`, `components/ProjetsEpargneClient.tsx:955,1132`, `components/RevenusTrackingClient.tsx:1030,1073,1111,1149`, `components/SavingsHistoryModal.tsx:40`, `components/AdhocIncomeModal.tsx:42`
**Rule:** `af-frontend-clean-code` § Accessibilite rule 1 — "HTML semantique — `<button>` pas `<div onClick>`"
**Problem:** Sheet backdrop `<div>` elements with `onClick` handlers have no `role`, no `tabIndex`, no keyboard event handlers. These are interactive elements that cannot be activated via keyboard. While backdrops are a secondary dismiss mechanism, the pattern is repeated 15+ times with no `role="button"` or keyboard alternative.
**Fix:** Either use `<button>` for the backdrop (styled with CSS), or add `role="presentation"` since the backdrop is a secondary dismiss mechanism and ensure Escape key closes the sheet (via `onKeyDown` on the sheet container).

## HIGH

### H1 — God Components: 5 components exceed 500 lines

**File:** `components/ProjetsEpargneClient.tsx` (1275L), `components/RevenusTrackingClient.tsx` (1196L), `components/Onboarding.tsx` (1113L), `components/ExpenseTemplateManager.tsx` (924L), `components/DepensesTrackingClient.tsx` (876L)
**Rule:** `af-frontend-clean-code` § Composants — "JSX > ~80 lignes: EXTRAIRE" + `af-clean-code` § Anti-patterns #2 God Object
**Problem:** These components combine data fetching orchestration, state management (10+ useState hooks each), multiple modals/sheets, list rendering, and form handling all in a single file. `ProjetsEpargneClient` at 1275 lines is the worst offender — it handles savings projects, debt management, extra payments, transfers, and history all in one component.
**Fix:** Extract into composable sub-components. For example, `ProjetsEpargneClient` should become: `SavingsProjectList`, `DebtList`, `ExtraPaymentSheet`, `ChargeDebtSheet`, each managing their own state. Move sheet state to the sub-component that owns it (per `af-frontend-clean-code` § State Management — "L'etat vit au plus bas niveau possible").

### H2 — Inline styles used everywhere instead of Tailwind (1355 `style={}` vs 315 `className=`)

**File:** All files in `components/`
**Rule:** `af-frontend-clean-code` § Styling Tailwind — "Inline styles pour ce que Tailwind couvre = a eviter"
**Problem:** The codebase has a 4.3:1 ratio of inline styles to className usage. Components like `AccueilClient.tsx`, `TabTableauDeBord.tsx`, `TabTimeline.tsx`, `TabSanteFinanciere.tsx`, `BottomNav.tsx`, and `MonthNavigator.tsx` use almost exclusively inline `style={}` objects for padding, margins, font sizes, colors, flexbox, borders — all of which Tailwind covers natively. This creates an inconsistent styling architecture: some components use Tailwind classes (via `className`), others use inline styles, and yet others mix both.
**Fix:** Migrate inline styles to Tailwind utility classes. Start with the most-used components (`AccueilClient`, `BottomNav`, tab components). Use CSS custom properties via Tailwind config for design tokens already defined in `globals.css`. Establish a convention: Tailwind for all styling, inline styles only for truly dynamic values (e.g., computed widths from data).

### H3 — Duplicated `@keyframes fadeInUp` across 4 files

**File:** `components/AccueilClient.tsx:281`, `components/accueil/TabTimeline.tsx:302`, `components/accueil/TabSanteFinanciere.tsx:645`, `components/accueil/TabTableauDeBord.tsx:351`
**Rule:** `af-clean-code` § DRY — "Pas de connaissance dupliquee" + `af-frontend-clean-code` § Styling Tailwind
**Problem:** The exact same `@keyframes fadeInUp` animation is defined in `<style>` tags inside 4 different components. This is both code duplication and a violation of Tailwind principles (animations belong in `globals.css` or `tailwind.config`).
**Fix:** Define `fadeInUp` once in `globals.css` or via Tailwind's `extend.keyframes` config. Remove all inline `<style>` tags.

### H4 — Duplicated icon/badge/status helper code across TabTimeline and ExpenseTrackingRow

**File:** `components/accueil/TabTimeline.tsx:62-108`, `components/ExpenseTrackingRow.tsx:14-81`
**Rule:** `af-clean-code` § DRY — "Pas de connaissance dupliquee"
**Problem:** `getIconVariant`, `iconStyles`, `getStatusBadge`, `getAmountColor` functions/maps are duplicated between these two files with nearly identical logic. Comments in `ExpenseTrackingRow.tsx` even say "same system as TabTimeline.tsx" and "same as TabTimeline.tsx", acknowledging the duplication.
**Fix:** Extract shared status/badge/icon utilities into a shared module (e.g., `lib/expense-display-utils.ts`) and import from both components.

### H5 — No `cn()` utility exists; no Tailwind class merging available

**File:** N/A (missing)
**Rule:** `af-frontend-clean-code` § Styling Tailwind — "`cn()` — indispensable"
**Problem:** The skill explicitly states `cn()` is "indispensable" for conditional Tailwind classes. The codebase has 0 usages of `cn()` and no such utility exists in `lib/`. Without it, conditional styling is either done via inline styles (current approach) or would require manual class concatenation prone to Tailwind conflicts.
**Fix:** Create `lib/cn.ts` with `clsx` + `tailwind-merge` wrapper. Use it for all conditional class logic as inline styles are migrated to Tailwind.

### H6 — Imperative DOM manipulation via onMouseEnter/onMouseLeave style mutations

**File:** `components/accueil/TabTableauDeBord.tsx:70-101` (SummaryCard), `components/MonthNavigator.tsx:37-48,96-111`, `components/AccueilClient.tsx:225-236`, `components/BottomNav.tsx:246-247,409-410`
**Rule:** `af-frontend-clean-code` § Styling Tailwind + § Anti-patterns
**Problem:** Multiple components manually mutate `style` properties via `onMouseEnter`/`onMouseLeave` event handlers (e.g., `e.currentTarget.style.borderColor = '...'`). This is imperative DOM manipulation that bypasses React's declarative model and Tailwind's hover variants (`hover:border-teal-700`, `hover:shadow-md`). It's fragile, verbose (8+ lines per hover effect), and impossible to maintain at scale.
**Fix:** Replace with Tailwind hover utilities: `hover:border-teal-700 hover:shadow-md hover:-translate-y-0.5 transition-all`. For CSS variable-based colors not in Tailwind config, extend the Tailwind theme.

### H7 — `LayoutShell` makes entire page tree a client component

**File:** `components/LayoutShell.tsx:1`, `app/layout.tsx:55`
**Rule:** `af-frontend-clean-code` § Server Components vs Client Components — "Descendre 'use client' au maximum"
**Problem:** `LayoutShell` is a `"use client"` component that wraps ALL page content (`{children}`) in `app/layout.tsx`. Since it uses `usePathname()` to decide whether to show the nav, the entire component tree under it becomes part of the client boundary. This means no page can leverage server component benefits for its layout wrapper. The `AuthProviders` wrapper above it is also `"use client"`, compounding the issue.
**Fix:** Move the conditional logic to middleware or use a layout group: `app/(auth)/` for auth pages (no nav) and `app/(app)/` for app pages (with nav). This way, the nav is a sibling, not a parent, and pages remain server components.

## MEDIUM

### M1 — No loading/error states for the home page server action calls

**File:** `app/page.tsx:44-64`
**Rule:** `af-frontend-clean-code` § Data Fetching — "Les 4 etats obligatoires: loading + error + empty + success"
**Problem:** The home page calls 9 server actions in `Promise.all` with no error handling. If any action fails, the entire page crashes. There is no Suspense boundary, no error boundary, and no try/catch.
**Fix:** Wrap the page in an error boundary. Consider using Suspense with fallback skeletons for the data-heavy sections.

### M2 — No Zod validation on form submissions — client-side only

**File:** `components/ExpenseModal.tsx:56-88`, `components/IncomeModal.tsx`, `components/ProjectModal.tsx`, `components/DebtModal.tsx`
**Rule:** `af-frontend-clean-code` § Forms — "Schema Zod unique dans shared/ — meme schema valide client ET serveur"
**Problem:** Forms validate with simple `isValid = name.trim() && amount && parseFloat(amount) > 0` instead of Zod schemas. There are no shared Zod schemas in `shared/` or `lib/schemas/`. Validation is ad-hoc per component with no contract between client and server.
**Fix:** Create Zod schemas for each form (e.g., `lib/schemas/expense.ts`), use them both in the server action and in the client form component via `zodResolver`.

### M3 — Multiple useState hooks manage related state — should use useReducer

**File:** `components/DepensesTrackingClient.tsx:68-81` (8 useState), `components/RevenusTrackingClient.tsx:42-49` (8 useState), `components/ProjetsEpargneClient.tsx:37-49` (13 useState)
**Rule:** `af-frontend-clean-code` § Hooks custom — "3+ useState + 1+ useEffect sur le meme concern → CREER un hook"
**Problem:** These components have 8-13 `useState` hooks managing related sheet/modal state (which sheet is open, which item is selected, input values). These are all part of the same "action sheet" concern.
**Fix:** Extract into a custom hook (e.g., `useActionSheet()`) or use `useReducer` with a discriminated union state type. This consolidates the state transitions and eliminates impossible states (e.g., two sheets open at once).

### M4 — Inline `<style>` tags for CSS that belongs in globals.css

**File:** `components/ExpenseModal.tsx:94-407`, `components/DepensesTrackingClient.tsx:776-809`
**Rule:** `af-frontend-clean-code` § Styling Tailwind — "Classes dans le JSX, pas de fichier CSS separe" + colocation
**Problem:** `ExpenseModal.tsx` has 313 lines of CSS in `<style>` tags — more CSS than JSX logic. `DepensesTrackingClient.tsx` has additional inline `<style>` for sheet components. These are scoped CSS blocks that could be Tailwind classes or, at minimum, moved to `globals.css`.
**Fix:** Migrate to Tailwind utility classes. For modal-specific patterns that recur across sheets, create reusable Tailwind-styled components (e.g., `<Sheet>`, `<SheetHandle>`, `<SheetHeader>`).

### M5 — `as any` type assertion suppresses type safety

**File:** `app/providers.tsx:14`
**Rule:** `af-clean-code` § Conventions TypeScript — type safety
**Problem:** `authClient as any` suppresses all type checking on the auth client prop. This hides potential API mismatches between the app's auth client and what `NeonAuthUIProvider` expects.
**Fix:** Investigate the actual type mismatch. If it's a version issue, use a more targeted assertion or add a type adapter.

### M6 — Comments that restate the code ("Section comments" smell)

**File:** `components/AccueilClient.tsx:83,193,243` (`// ====== MONUMENT: BALANCE HERO ======`, `// ====== TABS — UNDERLINE STYLE ======`), `components/DepensesTrackingClient.tsx:209,236,272,299` (similar section markers throughout)
**Rule:** `af-clean-code` § Commentaires — "Sections dans une longue fonction = smell → extraire"
**Problem:** Multiple components use `// ====== SECTION NAME ======` comments as section dividers. Per clean code principles, these are a code smell indicating the component should be extracted into sub-components rather than being divided by comments.
**Fix:** Extract each "section" into its own component. The comment `// ====== MONUMENT: BALANCE HERO ======` becomes a `<BalanceHero>` component, `// ====== TABS ======` becomes `<TabBar>`, etc.

### M7 — AccueilClient receives 12 props — data clump / prop explosion

**File:** `app/page.tsx:72-85`, `components/AccueilClient.tsx:41-53`
**Rule:** `af-frontend-clean-code` § Composants — "10+ props: passer aux compound components" + `af-clean-code` § Anti-patterns #5 Data Clump
**Problem:** `AccueilClient` receives 12 props, and then passes subsets of those props down to 3 tab sub-components. The server page also has a `Promise.all` with 9 calls that map 1:1 to these props. This is a data clump.
**Fix:** Group related props into typed objects (e.g., `dashboardData: DashboardData`, `healthData: HealthData`). Or better: have each tab component be a Server Component that fetches its own data, eliminating the prop waterfall entirely.

### M8 — `ChevronRight` component defined but never used

**File:** `components/accueil/TabTableauDeBord.tsx:19-28`
**Rule:** `af-clean-code` § Anti-patterns #7 Dead Code
**Problem:** The `ChevronRight` component is defined at the top of the file but never referenced anywhere in the JSX — the SVG chevron is inlined directly in the `SummaryCard` component instead.
**Fix:** Remove the unused `ChevronRight` component.

### M9 — No Escape key handler on modals/sheets

**File:** All sheet/modal components (15+ instances)
**Rule:** `af-frontend-clean-code` § Accessibilite rule 6 — "Navigation clavier — Tab, Enter, Escape"
**Problem:** None of the custom sheet/modal components handle the Escape key to close. Users navigating by keyboard have no way to dismiss these overlays without clicking the backdrop.
**Fix:** Add `onKeyDown` handler on the sheet container that listens for Escape and calls `onClose()`. Set `tabIndex={-1}` and auto-focus the sheet on open.

### M10 — No `aria-modal` or `role="dialog"` on any modal/sheet

**File:** All sheet/modal components
**Rule:** `af-frontend-clean-code` § Accessibilite rule 7 — "ARIA quand HTML ne suffit pas — role='dialog', aria-modal"
**Problem:** Custom sheet/modal overlays use plain `<div>` elements without `role="dialog"` or `aria-modal="true"`. Screen readers cannot identify these as modal dialogs, and focus is not trapped within them.
**Fix:** Add `role="dialog"` and `aria-modal="true"` to the sheet container. Add `aria-labelledby` pointing to the sheet title. Implement focus trapping.

### M11 — SVG icons inline throughout — no icon system

**File:** Throughout all components (50+ inline SVGs)
**Rule:** `af-frontend-clean-code` § Composants — "Reutilise 2+ fois → EXTRAIRE"
**Problem:** The same SVG icons (checkmark, clock, warning triangle, chevron, plus) are copy-pasted as inline SVGs across many components. The checkmark icon alone appears in at least 8 different files with identical markup.
**Fix:** Create an icon component library (e.g., `components/icons/`) or use an icon library like `lucide-react`. Each icon becomes a reusable component with consistent sizing and color props.

### M12 — Hardcoded color values bypass design token system

**File:** `components/accueil/TabTableauDeBord.tsx:36` (`#FFF7ED`, `#EA580C`), `components/accueil/TabTimeline.tsx:79` (`#ECFDF5`), `components/BottomNav.tsx:157-158` (multiple hardcoded hex colors), `components/ExpenseModal.tsx` (hardcoded rgba values)
**Rule:** `af-frontend-clean-code` § Styling Tailwind — "Design tokens: bg-primary > bg-blue-500. Jamais bg-blue-500 en dur"
**Problem:** Multiple hardcoded hex colors (`#FFF7ED`, `#EA580C`, `#ECFDF5`, `#10B981`, `#F0FDFA`, `#F1F5F9`) appear alongside CSS variable usage. This creates two styling systems and makes theme changes incomplete.
**Fix:** Add these colors as CSS custom properties in `globals.css` and reference them consistently via `var(--*)`.

## Systemic Issues

### S1 — Inline styles as the primary styling approach (architectural)

The codebase has 1355 inline `style={}` usages vs 315 `className=` usages. Despite having Tailwind v4 installed, the vast majority of UI is styled with inline style objects. This is not a one-off violation but a systematic architectural choice that contradicts the project's stated stack (Tailwind v4). This affects maintainability (no design system enforcement), theme-ability (inline styles bypass CSS variables in some cases), responsiveness (no easy breakpoint management), and hover/focus/active states (requires imperative DOM manipulation). This should be treated as a dedicated refactoring epic, not individual story fixes.

### S2 — No component decomposition strategy

The 5 largest components (1275L, 1196L, 1113L, 924L, 876L) are "God Components" that combine multiple concerns. This is a pattern, not an isolated case. The codebase would benefit from establishing a decomposition standard: Container components orchestrate, Presentational components render, Custom hooks encapsulate logic, and Shared UI components (Sheet, Modal, Card) provide reusable patterns.

### S3 — No shared form infrastructure

Every form in the application reinvents validation, submission, and error handling. There are no Zod schemas, no React Hook Form usage, no shared form components. Each modal builds its own form from scratch with local `useState` for each field.
