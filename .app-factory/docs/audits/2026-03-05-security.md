## HIGH

### H1 — No server-side input validation on any server action

**File:** `lib/actions/expenses.ts`, `lib/actions/incomes.ts`, `lib/actions/sections.ts`, `lib/actions/cards.ts`, `lib/actions/debts.ts`, `lib/actions/allocations.ts`, `lib/actions/settings.ts`, `lib/actions/onboarding.ts`, `lib/actions/monthly-expenses.ts`, `lib/actions/monthly-incomes.ts`, `lib/actions/debt-transactions.ts`
**Rule:** `af-security` § Validation des inputs
**Problem:** Zero Zod (or any other) validation exists in the entire codebase. No `zod` import anywhere. All server actions accept TypeScript types at the call boundary but perform no runtime validation. Server actions are publicly callable POST endpoints — a malicious client can send arbitrary JSON. TypeScript types are erased at runtime and provide zero protection. For example, `createExpense(data: CreateExpenseInput)` accepts `amount` as `number` but nothing prevents a client from sending `amount: "DROP TABLE"`, `amount: -999999`, or injecting unexpected fields.
**Impact:** Mass assignment risk (extra fields beyond the type could be passed), type confusion (strings where numbers expected), negative amounts, oversized strings. While SQL injection is mitigated by tagged template literals, business logic can be corrupted by invalid data.
**Fix:** Add Zod schemas for every server action input. Validate at the top of each action:
```typescript
const schema = z.object({
  name: z.string().min(1).max(255),
  amount: z.number().positive().max(999999),
  type: z.enum(['RECURRING', 'ONE_TIME', 'PLANNED']),
  // ...
});
const parsed = schema.parse(data);
```
This is the single most impactful fix in this audit. Every action with user input needs it.

### H2 — No rate limiting on any endpoint

**File:** `app/api/push/send/route.ts`, `app/api/push/subscribe/route.ts`, `app/api/cron/push/route.ts`, all server actions
**Rule:** `af-security` § Anti-patterns critiques (#6) + Checklist § Authentification
**Problem:** No rate limiting exists anywhere in the application. The push notification endpoints (`/api/push/send`, `/api/push/subscribe`) and all server actions can be called without any throttle. The login flow (handled by Neon Auth) may have its own rate limiting, but the application-level endpoints do not.
**Impact:** An authenticated attacker could spam push notifications, create thousands of expenses, or exhaust database connections. The `/api/push/send` endpoint is particularly concerning — a malicious user could trigger unlimited push notifications to their own subscriptions.
**Fix:** Add rate limiting middleware. For Vercel, use `@vercel/edge` or `upstash/ratelimit`:
```typescript
import { Ratelimit } from '@upstash/ratelimit';
import { Redis } from '@upstash/redis';
const ratelimit = new Ratelimit({ redis: Redis.fromEnv(), limiter: Ratelimit.slidingWindow(10, '1 m') });
```
Priority: `/api/push/send` and `/api/push/subscribe` first, then destructive server actions (`clearAllUserData`, `loadDemoData`).

### H3 — No middleware authentication guard — all pages accessible without auth

**File:** (missing) `middleware.ts`
**Rule:** `af-security` § Autorisation + Anti-patterns critiques (#4 — Client-side auth)
**Problem:** There is no `middleware.ts` file. Authentication is enforced only at the server action level via `requireAuth()`. The page components (server components in `app/`) call server actions that call `requireAuth()`, so the page will throw an error if the user is not authenticated. However, this means: (1) the page itself renders partially before the error, (2) there is no redirect to `/auth/sign-in`, (3) the user sees a raw error page, and (4) the landing page at `/landing` is intentionally public but there is no formal allow-list.
**Impact:** Poor UX for unauthenticated users hitting protected routes (unhandled error instead of redirect). No defense-in-depth — if a server action forgets `requireAuth()`, the page is completely open.
**Fix:** Add a `middleware.ts` at project root:
```typescript
import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';

const PUBLIC_PATHS = ['/auth', '/landing', '/api/cron', '/manifest.json', '/sw.js', '/icons'];

export function middleware(req: NextRequest) {
  const { pathname } = req.nextUrl;
  if (PUBLIC_PATHS.some(p => pathname.startsWith(p))) return NextResponse.next();
  const session = req.cookies.get('__session'); // adjust cookie name per Neon Auth
  if (!session) return NextResponse.redirect(new URL('/auth/sign-in', req.url));
  return NextResponse.next();
}
export const config = { matcher: ['/((?!_next/static|_next/image|favicon.ico).*)'] };
```

### H4 — Missing security headers in next.config.ts

**File:** `next.config.ts`
**Rule:** `af-security` § Headers de securite HTTP
**Problem:** The `next.config.ts` only defines `Cache-Control` headers for `sw.js` and `manifest.json`. None of the 5 essential security headers are configured: no `X-Frame-Options`, no `X-Content-Type-Options`, no `Strict-Transport-Security`, no `Referrer-Policy`, no `Permissions-Policy`, no `Content-Security-Policy`.
**Impact:** The app is vulnerable to clickjacking (no `X-Frame-Options`), MIME sniffing attacks (no `X-Content-Type-Options`), and has no CSP to mitigate XSS. Vercel adds some headers by default (HSTS), but the application should not rely on platform defaults for security.
**Fix:** Add security headers in `next.config.ts`:
```typescript
async headers() {
  return [
    {
      source: '/(.*)',
      headers: [
        { key: 'X-Frame-Options', value: 'DENY' },
        { key: 'X-Content-Type-Options', value: 'nosniff' },
        { key: 'Strict-Transport-Security', value: 'max-age=31536000; includeSubDomains' },
        { key: 'Referrer-Policy', value: 'strict-origin-when-cross-origin' },
        { key: 'Permissions-Policy', value: 'camera=(), microphone=(), geolocation=()' },
      ],
    },
    // ...existing sw.js and manifest.json headers
  ];
}
```

### H5 — `setAllocationSections` operates without user_id scoping

**File:** `lib/actions/allocations.ts:34-49`
**Rule:** `af-security` § Autorisation (IDOR prevention)
**Problem:** The helper function `setAllocationSections` deletes and inserts rows in `allocation_sections` using only the `allocation_id` — it does not verify that the allocation belongs to the current user. While the calling functions (`createAllocation`, `updateAllocation`) do verify ownership on the `income_allocations` table, there is no defense-in-depth on the junction table itself. Similarly, `fetchAllocationSections` (line 11-31) queries `allocation_sections` by allocation IDs without any user_id check.
**Impact:** If an attacker can guess or enumerate allocation UUIDs, they could potentially manipulate section links on allocations they don't own, though this requires the calling context to also be compromised.
**Fix:** Add a JOIN check in `setAllocationSections`:
```sql
DELETE FROM allocation_sections AS als
USING income_allocations AS ia
WHERE als.allocation_id = ia.id
  AND ia.id = ${allocationId}
  AND ia.user_id = ${userId}
```
Pass `userId` as a parameter to both helper functions.

## MEDIUM

### M1 — `SELECT *` used extensively — future column exposure risk

**File:** `lib/actions/expenses.ts:17-28` (e.*), `lib/actions/incomes.ts:12`, `lib/actions/sections.ts:11`, `lib/actions/cards.ts:11,18`, `lib/actions/settings.ts:10`, `lib/actions/expenses.ts:346` (savings_contributions), `app/api/push/send/route.ts:26`, plus 15+ `RETURNING *` statements
**Rule:** `af-security` § Protection des donnees (SELECT * rule)
**Problem:** Pervasive use of `SELECT *`, `e.*`, `d.*`, `me.*`, and `RETURNING *`. If a sensitive column is added to any table in the future (e.g., `internal_notes`, `admin_flag`), it will be automatically returned to the client without code changes.
**Impact:** Future data leakage risk. Currently no sensitive columns exist beyond what is intended, but this is a ticking time bomb.
**Fix:** Replace `SELECT *` with explicit column lists. For `RETURNING`, select only the columns needed. Start with tables most likely to get sensitive columns: `settings`, `push_subscriptions`.

### M4 — Push notification `/api/push/send` accepts arbitrary payload without validation

**File:** `app/api/push/send/route.ts:23`
**Rule:** `af-security` § Validation des inputs
**Problem:** The endpoint reads `const { title, body, url } = await req.json()` without any validation. The `url` parameter is passed directly to the notification payload — an attacker could inject a malicious URL (e.g., `javascript:alert(1)` or a phishing URL). The `title` and `body` could contain arbitrary strings of unlimited length.
**Impact:** An authenticated user could send themselves (and only themselves) a notification with a phishing URL. The browser notification `url` will be opened on click.
**Fix:** Validate with Zod:
```typescript
const sendSchema = z.object({
  title: z.string().max(100).optional(),
  body: z.string().min(1).max(500),
  url: z.string().url().startsWith('/').optional(), // only relative URLs
});
```

### M5 — Push subscription endpoint accepts unvalidated keys

**File:** `app/api/push/subscribe/route.ts:15-27`
**Rule:** `af-security` § Validation des inputs
**Problem:** The endpoint only checks `subscription?.endpoint` exists but does not validate the format of `endpoint` (should be a valid HTTPS URL), `p256dh`, or `auth` keys. Arbitrary strings are stored directly in the database.
**Impact:** Corrupted subscription data in the database. The `endpoint` should be a valid HTTPS push service URL.
**Fix:** Validate subscription fields:

```typescript
const subSchema = z.object({

```

### M6 — No Content-Security-Policy (CSP) header

**File:** `next.config.ts`
**Rule:** `af-security` § Headers de securite HTTP (CSP)
**Problem:** No CSP header is configured. CSP is the most powerful defense against XSS attacks. Without it, any injected script (from a compromised dependency, browser extension, or future XSS) can execute freely.
**Impact:** No XSS mitigation beyond React's default escaping. A financial app handling sensitive data should have CSP.
**Fix:** Add a basic CSP to `next.config.ts` headers. Start restrictive and relax as needed:
```
Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; connect-src 'self' https://*.neon.tech
```

### M7 — `clearAllUserData` and `loadDemoData` are destructive actions with no confirmation or audit trail

**File:** `lib/actions/demo-data.ts:377,29`
**Rule:** `af-security` § Logging de securite
**Problem:** `clearAllUserData()` permanently deletes ALL user data across 14 tables. `loadDemoData()` deletes existing sections and settings before inserting demo data. Neither action logs the event. There is no server-side confirmation mechanism — a single function call wipes everything.

### L3 — Timing-safe comparison not used for CRON_SECRET

**File:** `app/api/cron/push/route.ts:15`
**Rule:** `af-security` § Authentification
**Problem:** The CRON_SECRET comparison uses `!==` (strict equality), which is not timing-safe. A timing attack could theoretically extract the secret character by character.
**Impact:** Very low risk in practice (Vercel's network latency noise dwarfs timing differences), but it violates best practices.
**Fix:** Use `crypto.timingSafeEqual`:
```typescript
import { timingSafeEqual } from 'crypto';
const expected = Buffer.from(process.env.CRON_SECRET ?? '');
const actual = Buffer.from(authHeader?.replace('Bearer ', '') ?? '');
if (expected.length !== actual.length || !timingSafeEqual(expected, actual)) {
  return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
}
```

## Systemic Issues

### S1 — Complete absence of input validation layer

**Pattern:** All 11 server action files (40+ exported functions) accept user input with TypeScript types only. Zero runtime validation exists in the codebase. No Zod dependency installed.

**Recommendation:** Create a `lib/validators/` directory with Zod schemas. Each action file should import its schemas. This is the highest-priority systemic fix — it addresses H1, M4, M5, and L2 simultaneously.

### S2 — No security logging

**Pattern:** The application logs errors to `console.error` but never logs security-relevant events. No login attempts, no access denied events, no destructive operations, no rate limit hits.

**Recommendation:** Add structured security logging for: authentication events, authorization failures, destructive operations (`clearAllUserData`, `deleteExpense`, etc.), and suspicious activity patterns. Use a structured logger (pino) instead of `console.error`.

### S3 — `SELECT *` as default pattern

**Pattern:** 6 files use `SELECT *` or `e.*` for queries, and 15+ statements use `RETURNING *`. This is the default coding pattern, not the exception.

**Recommendation:** Establish a convention to always use explicit column lists. Add an ESLint rule or code review checkpoint for `SELECT *`.

## Positive Observations

These elements are correctly implemented:

1. **Authentication on all server actions:** Every exported server action (except `hasOrphanedData`) calls `requireAuth()` as its first operation.
2. **User scoping on all queries:** Every SQL query includes `AND user_id = ${userId}` (or `WHERE user_id = ${userId}`), preventing IDOR on data access.
3. **SQL injection protection:** All queries use Neon's tagged template literal (`sql\`...\``) which parameterizes automatically. No string concatenation in SQL.
4. **No XSS vectors:** No `dangerouslySetInnerHTML` anywhere. React's default escaping handles all output.
5. **No hardcoded secrets:** All secrets are accessed via `process.env`. No credentials in source code.
6. **NEXT_PUBLIC_ correctly used:** Only `NEXT_PUBLIC_VAPID_PUBLIC_KEY` is exposed — a public key, appropriate for client-side use.
7. **CSRF protection:** Server actions use POST by default. The auth route correctly patches `SameSite=None` to `SameSite=Lax`.
8. **Cron endpoint authentication:** `/api/cron/push` requires `Bearer ${CRON_SECRET}`.
9. **Env files properly gitignored:** `.env.local` and all `.env*.local` variants are in `.gitignore`.
10. **Soft delete pattern:** Expenses, incomes, debts use `is_active = false` instead of hard delete.
