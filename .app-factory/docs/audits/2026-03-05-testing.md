## Context

This is a personal finance PWA (alpha stage, single user) handling real money calculations, monthly budgets, income tracking, debt management, and savings. The codebase has **zero tests** — no unit tests, no integration tests, no E2E tests. The project has `@playwright/test` as a devDependency but no configuration or test files.

The risk is amplified by the financial nature of the application: calculation errors silently corrupt the user's budget view, leading to incorrect financial decisions.

## CRITICAL

### C1. Financial calculation functions have zero test coverage

**File:** `lib/utils.ts:93-107` (`calcMonthlyCost`), `lib/utils.ts:130-147` (`calcMonthlyIncome`), `lib/utils.ts:152-164` (`calcMonthlySuggested`)
**Rule:** `af-testing-strategy` § Coverage targets — "Logique metier critique: 90%+"
**Problem:** These three functions normalize financial amounts across frequencies (weekly, biweekly, monthly, quarterly, yearly). They use fractional multipliers (52/12, 26/12, 1/2, 1/3, 1/12) that are prone to floating-point rounding errors. They are called in 8+ locations across server actions and UI components. A bug here silently shows the user incorrect budget totals.
**Impact:** User sees wrong monthly budget amounts. Financial decisions based on incorrect data. No regression safety net.
**Fix:** Write unit tests with Vitest covering:
- Each frequency multiplier produces the expected monthly amount
- Edge cases: amount=0, very large amounts, fractional cents
- `calcMonthlyIncome` with VARIABLE frequency (estimated_amount vs null)
- `calcMonthlySuggested` with past target dates, zero remaining, exact boundary months

### C2. Biweekly pay date calculation is untested with complex date logic

**File:** `lib/utils.ts:174-213` (`countBiweeklyPayDatesInMonth`), `lib/utils.ts:220-247` (`getNextBiweeklyPayDate`)
**Rule:** `af-testing-strategy` § Principes fondamentaux — "Filet de securite (regressions)"
**Problem:** These functions contain non-trivial date arithmetic (14-day stepping forward/backward from an anchor date, month boundary detection). `countBiweeklyPayDatesInMonth` directly determines the expected monthly income amount in `generateMonthlyIncomes` (line 33-38 of `monthly-incomes.ts`). A bug means the user's expected income for a month is wrong (e.g., showing 2 paychecks when there are 3, or vice versa). The code has subtle complexity: comparison operators on Date objects, 86400000 magic number for day-to-ms conversion, and a fixed loop limit of 5 iterations.
**Impact:** Incorrect expected income amounts for biweekly-paid employees. This directly affects budget planning accuracy.
**Fix:** Write unit tests covering:
- Normal months (2 pay dates) vs "rich" months (3 pay dates)
- Various anchor dates (start of month, mid-month, end of month)
- February (28 days), months with 30/31 days
- Leap years
- `getNextBiweeklyPayDate` when anchor is in the past vs future vs today

### C3. Monthly expense generation logic is untested

**File:** `lib/actions/monthly-expenses.ts:13-82` (`calcDueDateForMonth`), `lib/actions/monthly-expenses.ts:85-246` (`generateMonthlyExpenses`)
**Rule:** `af-testing-strategy` § Decision Framework — "Unitaire: Logique pure isolee (calculs, validations, transformations)"
**Problem:** `calcDueDateForMonth` contains complex conditional logic for 6 frequency types (MONTHLY, BIMONTHLY, QUARTERLY, YEARLY, WEEKLY, BIWEEKLY) with skip-month guards. `generateMonthlyExpenses` orchestrates template-to-transaction conversion with spread_monthly logic, multiplier application, and debt payment generation. This is the core of the Template/Transaction pattern — the heart of the application. The `calcDueDateForMonth` function is pure (no DB dependency) and trivially testable, yet untested.
**Impact:** Expenses could be generated in wrong months, with wrong amounts, or duplicated/skipped. The spread_monthly feature divides amounts incorrectly without any verification.
**Fix:**
1. Extract `calcDueDateForMonth` to `lib/utils.ts` (it's already a pure function but trapped in a server action file)
2. Write exhaustive unit tests for `calcDueDateForMonth`:
   - MONTHLY: day 31 in February (should clamp to 28/29)
   - BIMONTHLY: alternating month skip pattern
   - QUARTERLY: 3-month intervals from reference date
   - YEARLY: only generates in the correct month
   - WEEKLY/BIWEEKLY: fallback to 1st of month
3. Write unit tests for spread_monthly amount calculation (QUARTERLY: /3, YEARLY: /12)

### C4. `calcNextDueDate` uses `new Date()` internally — untestable and untested

**File:** `lib/utils.ts:44-88`
**Rule:** `af-testing-strategy` § Anti-patterns — "#4 Dependance au temps" + `af-clean-code` § Testabilite — "new Date() dans la logique: Resultat imprevisible"
**Problem:** `calcNextDueDate` creates `new Date()` internally (line 48), making it non-deterministic. The function determines whether a due date is "past" relative to today, so its output changes daily. It handles 6 frequency types with date arithmetic. It is called by `createExpense` and `updateExpense` to set the `next_due_date` column. Currently impossible to test deterministically without mocking the system clock.
**Impact:** Cannot write reliable tests. Any test would be fragile and time-dependent. Meanwhile, the function has edge cases around month boundaries (day 31 in a 30-day month) and biweekly modular arithmetic that could silently produce wrong dates.
**Fix:**
1. Refactor to accept a `referenceDate` parameter: `calcNextDueDate(frequency, day, referenceDate = new Date())`
2. Write unit tests with fixed reference dates covering all 6 frequencies and edge cases

## HIGH

### H1. Server actions have no input validation

**File:** `lib/actions/expenses.ts:89-150` (`createExpense`), `lib/actions/debts.ts:38-64` (`createDebt`), `lib/actions/monthly-expenses.ts:455-467` (`updateMonthlyExpenseAmount`)
**Rule:** `af-testing-strategy` § Types de tests — "Unitaire: calculs, validations, transformations" + `af-clean-code` § Erreurs — "Fail fast: validation avant I/O"
**Problem:** Server actions accept arbitrary input without Zod validation. `createExpense` accepts `amount: number` but never validates it is positive. `updateMonthlyExpenseAmount` accepts `newAmount: number` with no bounds check — a negative amount would corrupt budget calculations. `createDebt` accepts `remaining_balance` and `payment_amount` without validating `remaining_balance >= 0` or `payment_amount > 0`.
**Impact:** Invalid data can be persisted to the database. Negative amounts would break all summary calculations.
**Fix:**
1. Add Zod schemas for all mutation inputs (createExpense, updateExpense, createDebt, etc.)
2. Write unit tests for the validation schemas (edge cases: 0, negative, very large numbers, missing fields)
3. Write component tests verifying that invalid input is rejected before reaching the DB

### H2. Savings transfer has no atomicity guarantee and no test

**File:** `lib/actions/expenses.ts:353-381` (`transferSavings`)
**Rule:** `af-testing-strategy` § Types de tests — "Component: Orchestration code + deps locales"
**Problem:** `transferSavings` performs 4 sequential SQL operations (debit contribution, debit balance, credit contribution, credit balance) without a transaction. If the process fails midway, money is debited from the source but not credited to the destination — a partial transfer that corrupts savings data. This is a financial integrity issue that also has zero test coverage.
**Impact:** Data corruption in case of partial failure. User loses tracked savings with no way to recover.
**Fix:**
1. Wrap the 4 SQL operations in a database transaction
2. Add validation: source `saved_amount >= amount` (prevent negative savings)
3. Write a component test that verifies both accounts are updated atomically

### H3. Debt payment flow lacks test coverage for cascading operations

**File:** `lib/actions/monthly-expenses.ts:327-368` (`markAsPaid`), `lib/actions/debts.ts:113-135` (`makeExtraPayment`), `lib/actions/debt-transactions.ts:8-49` (`addDebtTransaction`)
**Rule:** `af-testing-strategy` § Types de tests — "Component: verifier les effets de bord (etat DB)"
**Problem:** When a monthly debt expense is marked as paid, it triggers: (1) status update, (2) debt balance decrement, (3) auto-deactivation check, (4) transaction log insert. `addDebtTransaction` handles both PAYMENT and CHARGE types with opposite effects on balance. None of these cascading operations have any test coverage. A bug in the balance decrement formula (e.g., `GREATEST(remaining_balance - amount, 0)` clamping vs allowing negatives for charges) could silently corrupt debt tracking.
**Impact:** Incorrect debt balances, premature or missed auto-deactivation, lost transaction history.
**Fix:** Write component tests covering:
- markAsPaid with debt_id: verify balance decremented + transaction logged
- makeExtraPayment: verify balance decremented + auto-deactivation at zero
- addDebtTransaction CHARGE: verify balance incremented
- Edge case: payment larger than remaining balance (should clamp to 0)

**Rule:** `af-testing-strategy` § Principes fondamentaux — "Documentation vivante" + `af-clean-code` § DRY

## Systemic Issues

1. **Pure logic trapped in server action files.** Functions like `calcDueDateForMonth` are pure computation but live inside `"use server"` files, making them harder to import in tests. This is an architectural issue — pure business logic should be extracted to testable utility modules.

2. **No separation between business logic and data access.** Server actions mix validation, calculation, SQL queries, and cache invalidation in single functions. This makes unit testing impossible without mocking the database. The `af-clean-architecture` pattern of separating domain logic from infrastructure would dramatically improve testability.

3. **Financial calculations lack rounding strategy.** `Math.round(x * 100) / 100` is used inconsistently (only in `generateMonthlyExpenses` spread logic). Other calculations like `calcMonthlyCost` return raw floating-point results. For a financial application, a consistent rounding strategy should be tested and enforced.
