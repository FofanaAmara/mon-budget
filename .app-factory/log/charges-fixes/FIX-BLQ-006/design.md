# FIX-BLQ-006 — Design: Invalidate stale monthly_expenses on template update

**Story:** FIX-BLQ-006 — Modifying a template does not update already-generated monthly_expenses entries
**Level:** 2 (business logic)
**Scope:** backend
**Date:** 2026-03-05
**Migration:** None (no schema change)

---

## 1. Problem Statement

When a user modifies a recurring expense template (amount, frequency, spread_monthly, recurrence_day, is_active), the already-generated `monthly_expenses` entries for the current and future months are NOT updated. The `ON CONFLICT (expense_id, month) DO NOTHING` clause in `generateMonthlyExpenses()` preserves stale entries. The user sees no effect from their change until a new month generates a fresh entry.

## 2. Solution: Targeted Invalidation

**Strategy:** Delete-and-regenerate. After the UPDATE in `updateExpense()`, delete all eligible stale `monthly_expenses` entries. The next page visit triggers `generateMonthlyExpenses()` which recreates them with correct values via `ON CONFLICT DO NOTHING`.

This is simpler and safer than trying to UPDATE existing entries in place (which would require duplicating all the generation logic: spread_monthly calculation, frequency-based skip logic, due date computation).

## 3. Fields That Trigger Invalidation

| Field | Why it affects monthly_expenses |
|-------|--------------------------------|
| `amount` | Changes the monthly amount (direct or spread) |
| `recurrence_frequency` | Changes which months get entries and the multiplier |
| `spread_monthly` | Toggles between lump sum and divided amount |
| `recurrence_day` | Changes the due_date of generated entries |
| `is_active` | Deactivated template = no future entries |

**Fields that do NOT trigger invalidation:** `name`, `notes`, `section_id`, `card_id`, `auto_debit`, `currency`, `notify_*`, `reminder_offsets`. These are either display-only or do not affect the financial computation. (Note: `name`, `section_id`, `card_id`, `auto_debit`, `notes` ARE copied into monthly_expenses during generation, so stale entries will have outdated values for these fields too. However, these are cosmetic -- the financial impact is zero, and regeneration on the next page visit will fix them. Including these fields in the invalidation trigger list would cause unnecessary churn. If we discover users are confused by stale names, that is a separate discovery.)

## 4. The DELETE Query

```sql
DELETE FROM monthly_expenses
WHERE expense_id = $1
  AND user_id = $2
  AND status IN ('UPCOMING', 'OVERDUE')
  AND month >= $3  -- current month in "YYYY-MM" format
```

### Why these conditions

- **`expense_id = $1`**: Only touch entries generated from this template. Adhoc entries (`expense_id IS NULL`) and deferred copies (`expense_id IS NULL`) are untouched.

- **`user_id = $2`**: Ownership guard. Defense in depth -- `expense_id` alone would be sufficient since a template belongs to one user, but explicit user_id filtering is a security best practice.

- **`status IN ('UPCOMING', 'OVERDUE')`**: PAID entries are sacred -- they represent completed financial transactions (AC-4). DEFERRED entries are also excluded (see section 5).

- **`month >= $3`**: Only current and future months. Past months (even if UPCOMING/OVERDUE there) are not regenerated by `generateMonthlyExpenses()` since it only generates for the requested month. However, past UPCOMING/OVERDUE entries would be stale too. Decision: include them in the delete (`month >= current month`), because the user has no way to go back and regenerate them, and leaving stale entries in past months is confusing. The current month format is "YYYY-MM" and string comparison works correctly for this format.

### Current month computation

```typescript
const now = new Date();
const currentMonth = `${now.getFullYear()}-${String(now.getMonth() + 1).padStart(2, "0")}`;
```

This matches the existing month format used throughout the codebase (e.g., in `generateMonthlyExpenses`, `getMonthSummary`).

## 5. DEFERRED Entries -- Do NOT Touch

DEFERRED entries must be excluded from invalidation. Rationale:

1. When a user defers an expense (`deferExpenseToMonth`), the original entry is marked `DEFERRED` and a NEW entry is created in the target month with `expense_id = NULL`.
2. The DEFERRED entry is a historical record ("this was deferred from March to June"). Deleting it would erase the audit trail.
3. The replacement entry in the target month has `expense_id = NULL`, so it is not affected by our `WHERE expense_id = $1` clause anyway.
4. If the user changes the template after deferring, the deferred-from record stays as-is (it was already "closed"), and the deferred-to entry is independent.

The `status IN ('UPCOMING', 'OVERDUE')` condition naturally excludes DEFERRED entries. No additional logic needed.

## 6. Placement in updateExpense

```
updateExpense(id, data) {
  1. requireAuth()                          -- existing
  2. Recalculate next_due_date              -- existing
  3. UPDATE expenses SET ... RETURNING *    -- existing
  4. >>> NEW: Detect if invalidation-trigger fields changed <<<
  5. >>> NEW: If yes, DELETE stale monthly_expenses <<<
  6. revalidatePath(...)                    -- existing
  7. return rows[0]                         -- existing
}
```

### Detecting changed fields

Since `updateExpense` receives a `Partial<CreateExpenseInput>`, any field present in `data` is potentially changed. We do NOT need to compare old vs new values -- if the user submitted the field, we invalidate. This is a deliberate over-invalidation trade-off:

- **Pro:** Simpler code, no need to fetch the old row for comparison.
- **Pro:** Idempotent -- deleting + regenerating with the same values produces identical entries.
- **Con:** Unnecessary DELETE+INSERT cycle if the value didn't actually change.
- **Verdict:** The cost of a redundant DELETE+INSERT is negligible (one query, a few rows). The cost of a comparison miss (stale data persists) is a user-facing bug. Over-invalidation wins.

```typescript
const invalidationFields = ['amount', 'recurrence_frequency', 'spread_monthly', 'recurrence_day'] as const;
const hasInvalidationChange = invalidationFields.some(field => data[field] !== undefined);
```

**Special case: `is_active`** -- The `updateExpense` function does not handle `is_active` (deactivation goes through `deleteExpense` which sets `is_active = false`). Looking at the story's edge case about template deactivation: "If the user sets `is_active = false` on a template, UPCOMING/OVERDUE monthly_expenses for that template should be deleted." The current `deleteExpense` function only soft-deletes the template but does NOT clean up monthly_expenses. This is the same invalidation pattern. We should add it to `deleteExpense` as well.

However, `deleteExpense` does not receive the expense_id as a known template -- we need to add the DELETE query there too. This is within scope because AC-4's edge case explicitly requires it ("Template deactivation").

## 7. Implementation Plan

### File: `lib/actions/expenses.ts`

**Change 1 -- `updateExpense` function (after the UPDATE, before revalidatePath):**

```typescript
// Invalidate stale monthly_expenses if generation-affecting fields changed
const invalidationFields = ['amount', 'recurrence_frequency', 'spread_monthly', 'recurrence_day'] as const;
const hasInvalidationChange = invalidationFields.some(field => data[field] !== undefined);

if (hasInvalidationChange) {
  const now = new Date();
  const currentMonth = `${now.getFullYear()}-${String(now.getMonth() + 1).padStart(2, "0")}`;

  await sql`
    DELETE FROM monthly_expenses
    WHERE expense_id = ${id}
      AND user_id = ${userId}
      AND status IN ('UPCOMING', 'OVERDUE')
      AND month >= ${currentMonth}
  `;
}
```

**Change 2 -- `deleteExpense` function (after the UPDATE, before revalidatePath):**

```typescript
// Clean up stale monthly_expenses for the deactivated template
const now = new Date();
const currentMonth = `${now.getFullYear()}-${String(now.getMonth() + 1).padStart(2, "0")}`;

await sql`
  DELETE FROM monthly_expenses
  WHERE expense_id = ${id}
    AND user_id = ${userId}
    AND status IN ('UPCOMING', 'OVERDUE')
    AND month >= ${currentMonth}
`;
```

### File: `lib/actions/monthly-expenses.ts`

**No changes.** The existing `generateMonthlyExpenses()` function with `ON CONFLICT DO NOTHING` handles regeneration correctly after the stale entries are deleted.

## 8. Edge Cases Analysis

| Edge Case | Handling |
|-----------|----------|
| Multiple future months generated | The `month >= currentMonth` condition catches all of them. All UPCOMING/OVERDUE entries for current + future months are deleted. |
| Template deactivation (`is_active = false`) | Handled in `deleteExpense` (Change 2). UPCOMING/OVERDUE entries deleted. `generateMonthlyExpenses` won't recreate them because it filters on `is_active = true`. |
| Mid-month change after partial payment | PAID entries excluded by `status IN ('UPCOMING', 'OVERDUE')`. They remain untouched (AC-4). |
| Same-day toggle (spread on/off/on) | Each save triggers invalidation + regeneration on next page visit. Idempotent by design. Final state is always correct. |
| DEFERRED entries | Excluded by `status IN ('UPCOMING', 'OVERDUE')`. Historical audit trail preserved. |
| ONE_TIME expenses | `updateExpense` handles them too. If a ONE_TIME expense amount changes, its monthly_expense entry is invalidated and regenerated correctly (the `ON CONFLICT DO NOTHING` works on `expense_id + month`). |
| No monthly_expenses exist yet | DELETE on zero rows is a no-op. No error. |
| Concurrent page visit during invalidation | Worst case: `generateMonthlyExpenses` runs between the DELETE and the page render, recreating entries. Next render is also fine. No data corruption risk -- `ON CONFLICT DO NOTHING` is idempotent. |

## 9. Risks and Mitigations

| # | Risk | Severity | Mitigation |
|---|------|----------|------------|
| R1 | DELETE removes entries that should persist (e.g., manually adjusted amounts) | MEDIUM | The `status IN ('UPCOMING', 'OVERDUE')` guard protects PAID entries. Manually adjusted UPCOMING entries (via `updateMonthlyExpenseAmount`) will be lost and regenerated with template values. This is correct behavior -- the template is the source of truth. If the user adjusted a one-off amount AND then changes the template, the template change takes precedence. |
| R2 | Regeneration delay: between DELETE and next page visit, the dashboard shows incomplete data | LOW | The `revalidatePath` calls in `updateExpense` force a re-render of all relevant pages. Since `generateMonthlyExpenses` runs on page render of `/depenses` and `/`, the entries are recreated immediately on the next page load. The user never sees an empty state because the save action redirects to a page that triggers regeneration. |
| R3 | `month >= currentMonth` string comparison fails for year boundaries | NONE | "YYYY-MM" format ensures correct lexicographic ordering: "2025-12" < "2026-01". String comparison is safe. |
| R4 | Performance with many future entries | NONE | The DELETE query uses `expense_id` (indexed via the UNIQUE constraint on `(expense_id, month)`) + `user_id`. Even with 12+ entries, this is a single indexed DELETE. Negligible cost. |
| R5 | `deleteExpense` soft-deletes but we add a hard-delete of monthly_expenses | LOW | The monthly_expenses DELETE is scoped to UPCOMING/OVERDUE only. PAID entries (actual financial records) are preserved. The template itself is soft-deleted. No data loss for audit purposes. |

## 10. What This Design Does NOT Cover

- **Updating `name`, `section_id`, `card_id`, `notes`, `auto_debit` in existing monthly_expenses.** These cosmetic fields will be stale until the entry is regenerated. This is acceptable for now. If it becomes a problem, it is a separate improvement (not a blocker).
- **Invalidation for debt-linked monthly_expenses.** The story is specifically about expense templates. Debt payments have a separate generation path (`debt_id` not `expense_id`). Out of scope.
- **Past months.** Entries for months before the current month are NOT invalidated. They represent historical records. If they were UPCOMING/OVERDUE in a past month, the `autoMarkOverdue` function would have already processed them.
